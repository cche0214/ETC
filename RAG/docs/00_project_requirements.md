## 2. 实验基本要求
根据提供的高速公路 ETC 数据（CSV 格式），设计并实现一个 ETC 大数据管理平台，主页面展示内容自行设计，但必须满足以下核心技术指标：

### 2.1 基础功能
1.  **AI 数据扩充**：利用 AI 技术对原始数据进行扩充或增强，以满足更复杂的应用需求。
2.  **高并发模拟**：模拟真实车流环境，要求每秒产生 **50条** 数据。
3.  **数据预处理**：对原始脏数据进行清洗、格式化和标准化处理。
4.  **分层存储架构**：
    *   **方案一 (MySQL + 中间件)**：使用 MyCat 或 ShardingSphere + MySQL，需设计合理的分片逻辑（分库分表），并实现全局数据统计。
    *   **方案二 (HBase)**：使用 HBase 存储海量数据，需精心设计 RowKey 以支持高效的“Key-Value”查询和范围查询。
    *   *本项目采用了混合架构，同时实现了上述两种方案以发挥各自优势。*
5.  **交互式访问**：支持设置多种查询条件，对存储的数据进行灵活的交互式访问（本项目通过 SQL Agent 实现自然语言交互）。
6.  **实时可视化大屏**：设计可视化大屏，动态展示实时信息，要求刷新频率不低于 **每半分钟一次**。

### 2.2 扩展要求 (典型业务场景)
1.  **实时套牌车检测**：
    *   **流程**：Kafka 接收卡口数据 → Flink 窗口/状态计算 → 告警推送。
    *   **逻辑**：同一车牌在短时间内出现在地理距离不可达的两个不同卡口，即判定为套牌。
2.  **车流预警算法**：
    *   **流程**：Spark ML / TensorFlow 训练历史数据 → 生成预测模型 → 实时预测拥堵。
    *   **输出**：对收费站或路段的车流进行实时预警并显示。

## 3. 数据样例与字段含义
原始数据包含以下关键字段，用于描述一次过车事件：

| 字段 | 含义 | 备注 |
| :--- | :--- | :--- |
| `GCXH` | 过车序号 | 唯一标识符 |
| `XZQHMC` | 行政区划 | 如“睢宁县”、“铜山县”等 |
| `ROAD_ID` | 道路编号 | 如“G3”、“G2513” |
| `K_INDEX` | 桩号 | 标识具体位置，如“K731” |
| `GCSJ` | 过车时间 | 格式 `yyyy-MM-dd HH:mm:ss` |
| `HPZL` | 号牌种类 | 代码值，如“02”代表小型汽车 |
| `HPHM` | 号牌号码 | 车牌号，需脱敏处理 |
| `FXLX` | 行车方向 | 标识车辆行驶方向 |

## 4. 平台架构设计
平台采用经典的大数据流批一体架构：
*   **数据采集层**：Kafka
*   **计算层**：Flink (实时) + Spark/MySQL (离线/交互)
*   **存储层**：HBase (宽表) + MySQL (结构化分片) + Redis (缓存)
*   **应用层**：Vue.js (前端) + FastAPI (后端 Agent)
